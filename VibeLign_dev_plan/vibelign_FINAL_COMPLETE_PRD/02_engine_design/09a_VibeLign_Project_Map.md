
# VibeLign Project Map Specification
Version: Final Draft 1.0

This specification defines the structure and purpose of the internal
`.vibelign/project_map.json` file.

The Project Map is the structural "understanding layer" used by VibeLign
to analyze and reason about a codebase.

It allows AI-assisted tools to quickly understand:

- project structure
- module roles
- risky files
- entry points
- edit-safe areas

The Project Map is generated during:

vib init


--------------------------------------------------
Purpose
--------------------------------------------------

The Project Map provides a machine-readable representation of the project.

It is used by:

- vib doctor
- vib anchor
- vib patch
- vib explain
- vib guard

This allows all commands to operate using a shared understanding
of the project structure.


--------------------------------------------------
File Location
--------------------------------------------------

The Project Map is stored at:

.vibelign/project_map.json


--------------------------------------------------
Top-Level Structure
--------------------------------------------------

Example:

{
  "schema_version": 1,
  "project_name": "example_project",
  "entry_files": ["main.py"],
  "ui_modules": ["ui/window.py"],
  "core_modules": ["engine/patch.py"],
  "service_modules": ["services/auth.py"],
  "large_files": ["main.py"],
  "file_count": 38,
  "generated_at": "2026-01-01T12:00:00Z"
}


--------------------------------------------------
Fields
--------------------------------------------------

schema_version

Metadata schema version for compatibility checks.

project_name

Name of the root project folder.


entry_files

Files that appear to be application entry points.

Detected patterns:

- main.py
- app.py
- cli.py


ui_modules

Modules likely responsible for UI code.

Detected through:

- common UI imports
- folder names such as ui/, views/, widgets/


core_modules

Core engine or business logic modules.


service_modules

Service-layer modules such as:

- APIs
- data processing
- background services


large_files

Files exceeding recommended size thresholds.

Typical threshold:

300+ lines


file_count

Total number of source files detected in the project.


generated_at

Timestamp of map creation.


--------------------------------------------------
Metadata Ownership
--------------------------------------------------

Ownership rules:

- `vib init` is the primary writer of `.vibelign/project_map.json` in MVP.
- `vib doctor`, `vib anchor`, `vib patch`, `vib explain`, and `vib guard` are readers in MVP.
- The Project Map is observational metadata and must not directly modify source code.

Compatibility rules:

- `schema_version` is required.
- New optional fields may be added without breaking MVP readers.
- Removing or renaming existing required fields is a breaking change.
- Commands must fail clearly if the schema version is unsupported.


--------------------------------------------------
How the Map Is Used
--------------------------------------------------

vib doctor

Uses the map to detect:

- oversized files
- mixed responsibilities
- missing anchors


vib anchor

Uses the map to recommend anchor insertion locations.


vib patch

Uses the map to identify safe modification targets.


vib explain

Uses the map to prioritize important files when explaining changes.


vib guard

Uses the map to compare project structure before and after edits.


--------------------------------------------------
Safety Design
--------------------------------------------------

The Project Map must never modify project code.

It is purely observational metadata.


--------------------------------------------------
Regeneration
--------------------------------------------------

In MVP, the map is regenerated by rerunning `vib init`.

Dedicated refresh flags are post-MVP.


--------------------------------------------------
Design Principle
--------------------------------------------------

The Project Map should allow VibeLign to answer:

"What does this project look like structurally?"

within milliseconds.
